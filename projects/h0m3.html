H0M3

Production-ready home management for Spigot and Paper: multiple homes per player, inventory GUIs, polished teleport flow, and server-owner-friendly configuration—without sacrificing clarity in the codebase.

Spigot / Paper API Java 21+ Minecraft 1.21+ Open source
View on GitHub Releases

Why this project

Most servers need homes that scale beyond a single /sethome — players expect naming, limits, GUIs, and staff-tunable behavior. H0M3 is my answer: a cohesive plugin that treats homes as first-class data (metadata, timestamps, display settings) rather than a flat teleport map.

What I built

  • Full CRUD for homes — create, rename, move, delete, and teleport with permission gates at every step.
  • Dual GUIs — a grid of all homes plus a per-home management screen (material, lore, location, info).
  • Teleport pipeline — optional countdown, movement cancellation, arrival and countdown particles/sounds from config.
  • Messages & config — split config.yml / messages.yml with admin reload so copy can be iterated without recompiling.
  • Persistent storage — per-player data loaded on join and saved consistently across restarts.

Architecture

Command layer delegates to services for homes, effects, and messaging so gameplay rules stay testable and easy to extend (e.g. new subcommands or storage backends).

Operator experience

Length limits, max home counts, default GUI materials, and effect tuning are all data-driven—useful for networks that want strict prod/stage parity.

Technical notes

Built against the Paper ecosystem where it matters (modern event usage, safe teleport scheduling). Effects lean on the Bukkit particle/sound enums you already know, with validation-friendly names in YAML.

Lesson learned: separating player-facing strings from balance knobs early avoided a mess of magic numbers in Java and made the plugin approachable for non-developer server owners.
