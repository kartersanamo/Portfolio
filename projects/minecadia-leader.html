Minecadia Leader

A Discord bot for managing faction leaders, staff channels, and support workflows on the Minecadia Leader server—kitmap tickets, channel tooling, and leader-facing admin commands.

discord.py Python MySQL Live product
View on GitHub README

Overview

The Minecadia Leader Discord is where faction leadership and staff coordinate kitmap support. This bot keeps that server organized: ticket panels for bundle requests, tools to lock and reshape channels, and commands scoped to leaders and admins—not the same surface area as the main community bots.

What I built

  • Kitmap bundle tickets — ticket panel and toggle flow for faction bundle support.
  • Faction leader tools — leader utilities and admin list management for trusted roles.
  • Channel operations — lock, close, rename, purge, and edit commands for staff workflows.
  • Thread & request tooling — thread listing and staff request commands for faster triage.
  • Fleet-consistent architecture — cogs/, services/, repositories/, and ui/ aligned with other Minecadia bots.

Configuration

.env provides the Discord token and database credentials. assets/config.json holds presence and bot settings so operators can tune behavior without code changes.

Deployment

pip install -r requirements.txt, copy .env.example to .env, then python main.py or run.sh on the fleet host—restartable from the Bots dashboard.

Technical notes

Leader-server bots need tight permission boundaries: channel mutations and ticket flows must fail clearly when roles or channel types don’t match config, so staff aren’t left guessing why a command didn’t apply.

Takeaway: separating leader-facing and staff-facing command surfaces early keeps a small Discord from turning into a tangle of overlapping permissions.
